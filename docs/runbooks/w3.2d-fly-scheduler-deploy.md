# W3.2d — Fly Scheduled Machine Deploy Runbook (Operator Action)

Date: 2026-04-23
Status: repo artifacts shipped on `main` (Dockerfile, `services/ingest/fly.toml`). Live deployment requires operator steps below.

Against [`docs/superpowers/specs/2026-04-23-medevents-w3-2d-fly-scheduled-machines.md`](../superpowers/specs/2026-04-23-medevents-w3-2d-fly-scheduled-machines.md).

## What this runbook covers

Commands the **operator** runs to complete W3.2d. The assistant CANNOT run these autonomously because they require:

- A Fly.io account with billing enabled.
- `flyctl` CLI installed + authenticated on the operator's machine.
- Operator's choice of Postgres strategy (Fly Postgres cluster vs existing cloud Postgres).

Once these steps complete, the scheduled machine wakes hourly and runs `medevents-ingest run --all` end-to-end without further operator intervention. The admin UI's `/admin/sources` page will show `last_crawled_at` timestamps advancing automatically.

## 0 — Prerequisites

```bash
# 0.1 — Install flyctl (macOS)
brew install flyctl

# 0.2 — Sign in (opens a browser)
fly auth login

# 0.3 — Verify
fly auth whoami
```

A Fly account with a payment method on file is required — scheduled machines at 256MB RAM, one shared CPU, waking hourly, ~1 minute per run cost roughly **$0.50–$2.00/month**. The first `$5` of usage is free on all Fly accounts.

## 1 — Create the app

From the repo root (fly.toml lives there, not under `services/ingest/` — the build needs access to workspace-root `pyproject.toml` + `uv.lock`):

```bash
cd /path/to/medevents   # repo root
fly apps create medevents-ingest
# or a different name; if you pick a different name, update `app = "..."`
# in the repo-root fly.toml before step 5.
```

## 2 — Verify `uv.lock` is on `main`

Fly deploys use `uv sync --frozen --package medevents-ingest`, which requires the workspace-root `uv.lock` to be present and in sync with both `pyproject.toml` files (root + services/ingest). The lockfile is committed on `main` as of this wave — verify:

```bash
ls uv.lock
```

If the file is missing, regenerate + commit before deploying:

```bash
uv lock
git add uv.lock
git commit -m "chore(w3.2d): regenerate uv.lock"
git push
```

## 3 — Provision / attach Postgres

Pick ONE option.

### Option A — Fly Postgres (recommended for simplicity)

```bash
fly postgres create \
  --name medevents-postgres \
  --region iad \
  --vm-size shared-cpu-1x \
  --volume-size 1 \
  --initial-cluster-size 1
# Fly prints the DATABASE_URL to stdout — copy it for step 4.

fly postgres attach medevents-postgres --app medevents-ingest
# `attach` automatically sets DATABASE_URL as a Fly secret on the app
# and creates a role + database for it. Skip step 4 if you use attach.
```

After attach, run migrations against the Fly Postgres instance:

```bash
fly ssh console --app medevents-ingest -C "alembic -c ../../db/migrations/alembic.ini upgrade head"
# OR run alembic locally pointed at the proxied connection:
# fly proxy 5432:5432 --app medevents-postgres &
# DATABASE_URL=postgresql://...@localhost:5432/medevents uv run alembic -c db/migrations/alembic.ini upgrade head
```

### Option B — External cloud Postgres (Supabase / Neon / RDS)

Any Postgres reachable over public internet with SSL works. Make sure the provider gives you the SQLAlchemy-style DSN including `?sslmode=require`:

```
postgresql+psycopg://<user>:<pass>@<host>:5432/<db>?sslmode=require
```

Then run the migrations locally against that DSN (one-off, from the repo root):

```bash
DATABASE_URL="postgresql+psycopg://<user>:<pass>@<host>:5432/<db>?sslmode=require" \
  uv run alembic -c db/migrations/alembic.ini upgrade head
```

## 4 — Set Fly secrets (skip if Option A `attach` already did it)

```bash
fly secrets set \
  DATABASE_URL="postgresql+psycopg://<user>:<pass>@<host>:5432/<db>?sslmode=require" \
  --app medevents-ingest
```

Verify:

```bash
fly secrets list --app medevents-ingest
```

## 5 — Deploy

```bash
cd /path/to/medevents   # repo root — fly.toml is here, not in services/ingest/
fly deploy
```

The deploy:

1. Builds the image via `Dockerfile` (Fly runs the build remotely by default).
2. Pushes the image to Fly's registry.
3. Creates the scheduled machine per `fly.toml`'s `[[machines]]` block.

First deploy takes 3–5 minutes. Subsequent deploys on the same base layers take ~1 minute.

## 6 — Seed sources (one-off)

Against the same Postgres the machine will use, run the seed step:

```bash
# Option A — proxy to Fly Postgres:
fly proxy 5432:5432 --app medevents-postgres &
DATABASE_URL="postgresql+psycopg://medevents:<pass>@localhost:5432/medevents" \
  uv run --project services/ingest medevents-ingest seed-sources --path config/sources.yaml

# Option B — seed from the Fly machine itself:
fly ssh console --app medevents-ingest -C "medevents-ingest seed-sources --path /app/config/sources.yaml"
```

Verify the `sources` table has `ada` + `gnydm` rows with the expected `parser_name` values.

## 7 — Verify the first scheduled run

Scheduled machines wake on Fly's cron schedule. Wait up to an hour, or force a manual run for a faster smoke test:

```bash
fly machine run <image-ref> \
  --app medevents-ingest \
  --schedule none \
  --command "medevents-ingest run --all"

# OR list the scheduled machine and manually start it:
fly machines list --app medevents-ingest
fly machine start <machine-id>
```

Tail logs:

```bash
fly logs --app medevents-ingest
```

Expected first-run output (paraphrased):

```
source=ada fetched=2 skipped_unchanged=0 created=6 updated=0 review_items=0
source=gnydm fetched=2 skipped_unchanged=0 created=3 updated=1 review_items=0
batch=run-all sources=2 succeeded=2 failed=0 skipped_not_due=0
```

## 8 — Verify the admin UI (optional but recommended)

If you've also deployed the Next.js admin UI, navigate to `/admin/sources`. Both `ada` and `gnydm` should show a real `last_crawled_at` timestamp from the Fly run (step 7). Not null — the admin UI was reading empty timestamps before W3.2a; W3.2d lets Fly write them autonomously.

## 9 — Monitor + rollback

**Monitoring (MVP-level, Fly-native):**

```bash
fly logs --app medevents-ingest              # tail the latest run
fly machines list --app medevents-ingest     # see next scheduled wake time
fly releases list --app medevents-ingest     # image history
```

**Rollback a bad deploy:**

```bash
fly releases list --app medevents-ingest
fly releases rollback <release-id> --app medevents-ingest
```

**Pause the scheduled machine (during incidents):**

```bash
# Stop the machine from waking. Resume via `fly machines start`.
fly machines stop <machine-id> --app medevents-ingest
```

Or toggle individual sources via the admin UI's source detail page (`is_active = false`). The Fly machine continues to wake, but inactive sources are skipped (W3.2b §4 D3).
