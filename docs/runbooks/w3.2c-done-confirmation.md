## Notes on edge behavior to watch in production

- **Field clobber** from rare upstream genuine-removal is now impossible via the ingest path. Operator must clear manually via the admin edit form. This is the right asymmetry for ingest-as-inbound-side-of-operator-curated-surface; the admin UI's edit form already supports per-field clears.
- **Drift alerting** is still review-queue-based — no pager integration. If a detail page goes silent, it lands in `/admin/review` as a `parser_failure` with `details_json.page_kind='detail'`; operator triages from there. W3.2c doesn't add alerting infrastructure; that's explicitly out of scope.
- **Third-source (W3.2d) now inherits** the correct classifier contract: listing-zero AND detail-zero both fire review items. The third source's author doesn't need to remember to add detail-drift handling; it's in the pipeline.

## Residual risks (non-blocking)

- **Partial-page hash change** under detail-only-changed scenario: the homepage-detail path might still mask drift if the FORM of detail parsing changes (different HTML structure that still yields some events but with wrong content). The drift signal only fires on ZERO events; partial-wrong-content is another class of drift W3.2c doesn't address. Follow-up candidate: content-diff assertions on key fields (e.g. the 2026 event's starts_on must fall in a plausible year window).
- **Explicit-clear mechanism** not implemented. Operators who want to programmatically clear fields via ingest would need a follow-up wave introducing a `CLEAR` sentinel on `ParsedEvent` fields or a `cleared_fields: set[str]` property. Wait for a real request before adding.
