## Plan deviations (documented for audit)

- **`session.rollback()` inside `run_all`'s exception handler.** Not in the plan as written, but necessary because W3.2a's fresh-session error bookkeeping leaves the outer session in psycopg's "current transaction is aborted" state after the first per-source failure. Without the rollback, subsequent sources in the batch hit `InFailedSqlTransaction`. The "continues after failure" integration test locks this behavior. Watch-out item #4 in the Task 3 dispatch flagged this risk; the subagent caught it during test runs and added the rollback.

## Residual risks (non-blocking; carry to W3.2c+)

- **Detail-page drift still silent** — spec §7 out-of-scope; this is precisely what W3.2c addresses. If the GNYDM homepage classifier silently yields zero events, `run --all` will report the source as succeeded with 0 created/updated — no signal.
- **`run --all --dry-run` still exits 4** — dry-run is orthogonal scope, deferred to W3.3+. Operators who need preview runs for a batch must run sources individually.
- **Per-source sequential execution** — two sources run fast; if we get to ten sources with slow network, parallel-within-batch becomes worth considering. Not urgent; Fly scheduled machines wake hourly, which dwarfs any sequential-batch duration up to tens of minutes.
- **30-day "monthly" is an approximation.** Operators who expect calendar-aligned monthly runs (e.g. "the 1st of every month") need a new `crawl_frequency` value + migration. Not urgent; `monthly = 30 days` is documented behavior.
