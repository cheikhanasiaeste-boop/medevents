## §1. Scope closed

The `medevents-ingest run --dry-run` flag — which previously stubbed `exit(4)` with "not yet implemented (W3.2+)" — now previews exactly what a real run would do, with zero DB writes at any boundary. Works under both `--source CODE` and `--all`, freely composable with `--force`.

## §2. Contract compliance (spec §3)

### §3.1 CLI surface — PASS

- `run --source CODE --dry-run` valid; `run --all --dry-run` valid; `run --all --force --dry-run` valid. Mutex rules unchanged (`--source` xor `--all`; exactly one required).
- Exit codes: 0 on success, 1 under `--all --dry-run` when every selected source failed the preview, 2 on usage error (source not found, mutex violation), 3 on unknown parser.

### §3.2 Output shape — PASS (live evidence §5)

- Per-page: `dry_run source=CODE page=URL kind=listing|detail status=STATUS` where STATUS ∈ {`would_fetch_and_parse`, `would_skip_unchanged`, `would_file_review_item_source_blocked`, `would_file_review_item_parser_failure`}.
- Per-candidate-event: `dry_run source=CODE action=would_create|would_update title="..." starts_on=ISO city=CITY venue="..."`.
- Source summary: `dry_run=1 source=CODE fetched=N skipped_unchanged=N created=N updated=N review_items=N`.
- Batch summary (under `--all`): `dry_run=1 batch=run-all sources=N succeeded=N failed=N skipped_not_due=N`.

### §3.3 Write-path bypass — PASS (unit + integration coverage)

All 8 write sites bypassed under `dry_run=True`:

1. `update_source_run_status` (success + error paths) — `pipeline.py:160`, via `_record_error_bookkeeping_fresh_session` gate at `pipeline.py:143` + `pipeline.py:153`
2. `_record_error_bookkeeping_fresh_session` — gated at `pipeline.py:143` + `pipeline.py:153`

`cli.py:76-83` and `cli.py:108-113` both wrap the pipeline call in `try: ... finally: if dry_run: s.rollback()` so that any write that slipped past the flag threading is undone before `session_scope`'s implicit commit. Independent defense from the bypass gates.

## §6. Durable findings

1. **Spec §4 D5 compliance required a follow-up fix.** The initial Task 1 implementation used a zero-UUID sentinel for `source_page_id` under dry-run, which made `get_last_content_hash(session, source_page_id)` always return None → `would_skip_unchanged` never fired. Integration test 13 caught this (xfail → PASS after `214389e` added `get_last_content_hash_by_url(session, *, source_id, url)`). **Lesson:** when a spec decision says "the read should still work", integration tests that exercise the read are load-bearing — unit tests alone won't catch a schema-mismatched stub.

2. **Belt-and-braces rollback is a cheap, valuable defense.** The explicit `try: ... finally: if dry_run: s.rollback()` in cli.py costs two lines per call site and covers the "one missing bypass branch" failure mode independent of the bypass gates. Future refactors that add new write sites will still be safe even if the author forgets the `if not dry_run:` guard — the test suite will catch the correctness issue, but the runtime will contain the blast radius.

3. **`testseed` leftover in dev DB.** A stale seed row from prior test runs leaked into the live `sources` table; the `--all` dry-run surfaced it (UnknownParserError). Future cleanup task: audit what leaves leftovers in dev DB and add idempotent cleanup or stricter TRUNCATE scope in tests. Tracked as a small item in TODO.md.

4. **Preview output shape scales poorly for very-large sources.** A source with 500 candidates would print 500+ lines. Current output is terse enough for the 3 live sources (8-20 lines each) but a future `--dry-run --summary-only` flag could be useful. Deferred until the need materializes.

5. **Content-hash read is by `(source_id, url)` not by surrogate `id`.** Discovered during D5 follow-up: `source_pages` has a `UNIQUE(source_id, url)` constraint and stores `content_hash` directly on the row (no `source_page_fetches` table). So by-URL lookup is a single SELECT — no join needed. This is relevant for any future read-only feature that needs to reach source-page state without writing.
